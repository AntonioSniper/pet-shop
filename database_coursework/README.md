# database_coursework

Эта папка содержит материалы по базе данных для существующего Laravel-проекта **pet-shop**. Основной проект, таблицы, миграции и файл `database/database.sqlite` здесь не пересоздаются и не изменяются.

Материалы описывают текущую структуру БД pet-shop:

- `users`
- `categories`
- `products`
- `orders`
- `order_items`

## Назначение базы данных

База данных pet-shop хранит данные интернет-магазина товаров для животных. В ней есть пользователи, категории каталога, товары, заказы и состав каждого заказа. Такая структура позволяет вести каталог товаров, оформлять покупки, хранить историю заказов пользователя и рассчитывать итоговую стоимость заказа.

## Таблицы и поля

### users

Таблица хранит зарегистрированных пользователей магазина.

| Поле | Назначение |
| --- | --- |
| `id` | первичный ключ пользователя |
| `name` | имя пользователя |
| `email` | уникальный адрес электронной почты |
| `email_verified_at` | дата подтверждения email, может быть пустой |
| `password` | хеш пароля |
| `remember_token` | токен Laravel для функции "запомнить меня" |
| `created_at` | дата создания записи |
| `updated_at` | дата последнего обновления |
| `role` | роль пользователя, например `user` или `admin` |

Первичный ключ: `users.id`.

### categories

Таблица хранит категории товаров.

| Поле | Назначение |
| --- | --- |
| `id` | первичный ключ категории |
| `name` | название категории |
| `slug` | уникальный человекочитаемый идентификатор для URL |
| `description` | описание категории |
| `created_at` | дата создания записи |
| `updated_at` | дата последнего обновления |

Первичный ключ: `categories.id`.

### products

Таблица хранит товары магазина.

| Поле | Назначение |
| --- | --- |
| `id` | первичный ключ товара |
| `category_id` | внешний ключ на категорию товара |
| `name` | название товара |
| `slug` | уникальный человекочитаемый идентификатор для URL |
| `description` | описание товара |
| `price` | текущая цена товара |
| `stock` | остаток на складе |
| `image` | путь или имя изображения товара |
| `is_active` | признак доступности товара в каталоге |
| `created_at` | дата создания записи |
| `updated_at` | дата последнего обновления |

Первичный ключ: `products.id`.

Внешний ключ: `products.category_id` ссылается на `categories.id`.

### orders

Таблица хранит общую информацию о заказе.

| Поле | Назначение |
| --- | --- |
| `id` | первичный ключ заказа |
| `user_id` | внешний ключ на пользователя, оформившего заказ |
| `total_amount` | итоговая сумма заказа |
| `status` | статус заказа |
| `delivery_address` | адрес доставки |
| `phone` | телефон покупателя |
| `comment` | комментарий к заказу |
| `created_at` | дата создания заказа |
| `updated_at` | дата последнего обновления |

Первичный ключ: `orders.id`.

Внешний ключ: `orders.user_id` ссылается на `users.id`.

### order_items

Таблица хранит позиции заказа: какие товары, в каком количестве и по какой цене вошли в конкретный заказ.

| Поле | Назначение |
| --- | --- |
| `id` | первичный ключ позиции заказа |
| `order_id` | внешний ключ на заказ |
| `product_id` | внешний ключ на товар |
| `quantity` | количество товара в заказе |
| `price` | цена товара на момент оформления заказа |
| `subtotal` | сумма позиции заказа |
| `created_at` | дата создания записи |
| `updated_at` | дата последнего обновления |

Первичный ключ: `order_items.id`.

Внешние ключи:

- `order_items.order_id` ссылается на `orders.id`.
- `order_items.product_id` ссылается на `products.id`.

## Связи между таблицами

- `users` 1:N `orders` - один пользователь может оформить много заказов.
- `categories` 1:N `products` - одна категория может содержать много товаров.
- `orders` 1:N `order_items` - один заказ может включать много позиций.
- `products` 1:N `order_items` - один товар может встречаться во многих позициях заказов.

## Зачем нужны orders и order_items

Таблица `orders` описывает сам заказ как документ: покупателя, статус, адрес доставки, телефон, комментарий и итоговую сумму.

Таблица `order_items` описывает состав заказа. Один заказ может содержать несколько товаров, поэтому позиции нельзя удобно хранить прямо в `orders`. Отдельная таблица позволяет хранить любое количество товаров в заказе, фиксировать количество и цену товара на момент покупки.

Такой подход нужен, потому что цена товара в `products` может измениться позже, а история заказа должна оставаться корректной.

## Нормализация и 3НФ

Основная логическая структура приведена к третьей нормальной форме:

- каждая таблица описывает одну сущность: пользователь, категория, товар, заказ или позиция заказа;
- у каждой сущности есть первичный ключ;
- повторяющиеся группы вынесены в отдельные таблицы, например состав заказа хранится в `order_items`;
- сведения о категории не дублируются в товарах, вместо этого используется `products.category_id`;
- сведения о пользователе не дублируются в заказах, вместо этого используется `orders.user_id`;
- связь товаров и заказов реализована через `order_items`, потому что это отношение "многие ко многим" между заказами и товарами.

Поля `orders.total_amount` и `order_items.subtotal` являются вычисляемыми суммами. В строгой теории нормализации такие поля можно считать контролируемой денормализацией для удобства отображения и ускорения отчетов. В материалах добавлены триггеры, которые пересчитывают `orders.total_amount` по позициям заказа и помогают поддерживать согласованность данных.

## Резервное копирование

Файл `backup_sqlite.bat` делает резервную копию текущей SQLite-базы:

1. определяет корень проекта относительно папки `database_coursework`;
2. проверяет наличие `database/database.sqlite`;
3. создает папку `backups`, если ее еще нет;
4. копирует файл БД в `backups`;
5. использует имя вида `petshop_backup_YYYYMMDD.sqlite`.

Скрипт не запускает миграции и не изменяет таблицы. Он только копирует существующий файл базы данных.

Пример запуска из корня проекта:

```bat
database_coursework\backup_sqlite.bat
```

## Восстановление

Файл `restore_sqlite.bat` восстанавливает SQLite-базу из выбранного backup-файла. Скрипт принимает имя файла из папки `backups` или полный путь к файлу.

Примеры:

```bat
database_coursework\restore_sqlite.bat petshop_backup_20260509.sqlite
database_coursework\restore_sqlite.bat C:\path\to\petshop_backup_20260509.sqlite
```

При восстановлении текущий файл `database/database.sqlite` заменяется выбранной копией. Перед восстановлением желательно сделать свежий backup.

## Дополнительные SQL-файлы

- `seed_extra_data.sql` добавляет дополнительные категории и товары на русском языке. Используется `INSERT OR IGNORE`, поэтому существующие записи не удаляются.
- `procedure_sqlite.sql` содержит аналог процедуры `GetOrderSummaryByPeriod` для SQLite в виде параметризованного `SELECT`-запроса с параметрами `:start_date` и `:end_date`.
- `trigger_sqlite.sql` создает триггеры для пересчета `orders.total_amount` после вставки, изменения и удаления строк в `order_items`.

## Анализ данных

Файл `analysis_module.py` подключается к базе через SQLAlchemy. Путь к БД хранится в `db_config.json` и указывает на существующий файл `database/database.sqlite`.

Модуль умеет:

- выводить список заказов с полями `order_id`, `user_name`, `created_at`, `status`, `total_amount`;
- сортировать заказы по доступным колонкам;
- фильтровать заказы по пользователю или показывать все;
- искать строку по заказам и помечать совпадения меткой `[MATCH]`;
- считать статистику: всего заказов и общую сумму заказов;
- обрабатывать ошибки отсутствия БД, SQL-ошибки, пустой результат и неправильный ввод.

Команды для запуска:

```bash
pip install sqlalchemy
python database_coursework/analysis_module.py
```
